# Project Completion Status & Remaining Tasks

## Overview
The Developer WhatsApp Assistant is a hybrid Node.js (WhatsApp transport) + Python (LLM brain) application with intent routing, stateful memory, and delivery execution. This plan documents what remains to complete the project based on the current state.

## Current State (What's Already Implemented)

### ✅ Core Infrastructure
- **Python FastAPI bridge** (`app/api/bridge.py`) - HTTP endpoint for Node→Python communication
- **DevAssistantBrain** (`app/assistant.py`) - Orchestrates LLM intent classification + delivery
- **DeliveryService** (`app/delivery/service.py`) - Dispatches intents to handlers with plain-language responses
- **DatabaseManager** (`app/database.py`) - SQLite with users, messages, intent_logs, scheduled_tasks, expense_logs, code_runs
- **LLMProvider** (`app/llm/providers.py` + `src/llm-provider.js`) - Unified OpenAI/Anthropic/Google interface
- **WhatsApp Bot** (`src/whatsapp-bot.js`) - Baileys-based transport with QR auth
- **Bridge Client** (`src/bridge-client.js`) - Node→Python HTTP communication
- **Reminder Scheduler** (`app/scheduler.py`) - Polls due tasks and delivers via callback
- **Configuration** (`app/config.py`) - Pydantic settings with env file support

### ✅ Intent Handlers (All 6 Intents)
1. **schedule_task** - Parses time, stores tasks, handles incomplete metadata with clarification
2. **log_expense** - Extracts amount/currency/category, persists to DB
3. **execute_code** - Python sandbox with timeout, blocked patterns, output capture
4. **debug_code** - LLM-based debugging analysis
5. **summarize_link** - URL fetch with content-type validation + text summarization
6. **general_chat** - Conversational responses with context

### ✅ Tests
- `tests/test_delivery.py` - Comprehensive delivery handler tests (158 lines)
- `tests/test_database.py` - Database lifecycle tests
- `tests/test_delivery.py` covers: schedule (complete/incomplete/invalid), expense, code execution (success/blocked/wrong-lang), summarize (text/url/invalid), debug, unknown intent, general_chat

### ✅ Architecture Decisions
- **Bridge mode** (`npm run start:bridge`) - Full feature parity via Python delivery (PRIMARY PATH)
- **Legacy mode** (`npm run start:legacy`) - Intent classification only (returns warning about limitations)
- Plain-language responses (no raw JSON to users)
- Silent catch for invalid LLM JSON → graceful fallback

## Remaining Tasks to Complete the Project

Based on the three existing plan files, here are the remaining gaps ordered by priority:

### Priority 1: Harden Delivery Execution (Critical)

#### 1.1 Complete `test_delivery.py` Incomplete Cases
- **Status**: Tests exist but may need expansion for edge cases
- **Actions**:
  - Add tests for `general_chat` with actual LLM mock responses
  - Add timeout/output truncation tests for `execute_code`
  - Add fallback text behavior tests (invalid classifier output)
  - Ensure all delivery paths have >90% coverage
- **Validation**: `npm run python:test` passes

#### 1.2 Harden `summarize_link` Fetch Safeguards
- **Status**: Basic validation exists (HTTP/HTTPS check, content-type, timeout=10s, 10KB cap)
- **Gaps**: 
  - No maximum bytes read limit on urllib (could read huge HTML into memory)
  - No redirect limit
  - No SSL verification bypass option
  - No user-agent rotation / rate limiting
- **Actions**:
  - Add `max_bytes` read limit (e.g., 50KB) before decode
  - Add redirect limit (max 5)
  - Add timeout on read operation (not just connection)
  - Validate content-length header before fetch
- **Tests**: Mock timeout, oversized content, redirect loops

#### 1.3 Harden `execute_code` Sandbox Constraints
- **Status**: Basic blocklist (os.system, subprocess, eval, exec, open, imports), timeout=5s, output cap 2KB
- **Gaps**:
  - No memory limit (can allocate GB via list comprehensions)
  - No CPU limit beyond timeout (can spin in Python)
  - Blocked patterns can be bypassed (e.g., `__import__('os')`)
  - No seccomp/container isolation
  - Network access not blocked (can make requests)
- **Actions**:
  - Use `resource` module to set RLIMIT_AS (address space) and RLIMIT_CPU
  - Add `ulimit` style constraints before subprocess
  - Expand blocklist to include `__import__`, `getattr`, `globals`, `locals`
  - Block network via iptables or run in network namespace (advanced)
  - Add deterministic timeout using `signal.alarm` as backup
  - Log security events separately
- **Tests**: Memory exhaustion, CPU spin, import bypass attempts

#### 2.2 Add Task Management Intents

### Priority 3: Scheduler Hardening

#### 3.1 Add Retry/Backoff Policy
- **Status**: Failed deliveries marked pending and retried on next poll (no limit)
- **Gaps**: No exponential backoff, no max attempts, no dead-letter state
- **Actions**:
  - Add `attempt_count` column to scheduled_tasks (default 0)
  - Add `max_attempts` column (default 3) or constant
  - In `_deliver()`: on failure, increment attempt_count
  - If attempt_count >= max_attempts: mark status='failed' (dead-letter)
  - Implement exponential backoff: delay = min(2^attempt_count * base_delay, max_delay)
  - Store `last_error` and `last_attempt_at`
- **Tests**: Verify failed task retries, eventual failure to dead-letter

#### 3.2 Secure Callback Endpoint
- **Status**: `/send` callback has no authentication
- **Gaps**: Anyone can trigger fake reminders
- **Actions**:
  - Add `SCHEDULER_CALLBACK_SECRET` to settings
  - In `bridge.py` `/send` endpoint: verify HMAC signature header
  - Sign payload in scheduler: `HMAC_SHA256(payload_json, secret)`
  - Verify in callback before processing
  - Alternatively: use shared token in Authorization header
- **Tests**: Reject requests without/with invalid signature

#### 3.3 Add Delivery Idempotency
- **Status**: Race condition possible if scheduler runs twice before state update
- **Gaps**: Same task could be delivered multiple times
- **Actions**:
  - Add `delivery_idempotency_key` or use task ID + delivery timestamp
  - In `_deliver()`: check if already delivered in last N seconds
  - Use Redis or DB row locking (SELECT FOR UPDATE)
  - Or simpler: atomic UPDATE with status check in single query
- **Tests**: Concurrent delivery attempts
