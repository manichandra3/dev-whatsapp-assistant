# Delivery Layer Implementation Plan

## Goal
Implement the missing "delivery" stage so parsed intents are not only classified/logged, but also executed and replied to with useful user-facing output.

## Current State (confirmed)
- Intent classification is implemented in both Python and Node stacks (`app/assistant.py`, `src/index.js`).
- LLM prompt currently forces a strict JSON classifier output only (`app/llm/providers.py`, `src/llm-provider.js`).
- Parsed intents are stored, but no intent-specific action handlers are executed.
- WhatsApp currently receives raw JSON as message text instead of executed outcomes.

## Target Behavior
- Keep intent classification as step 1.
- Add delivery execution as step 2 based on intent.
- Return a clear plain-language response to WhatsApp users (not raw intent JSON).
- Persist relevant artifacts for reminders/expenses/executions.

## Implementation Plan

### 1) Add delivery orchestration in Python brain (primary path)
- Create a dedicated delivery service module (for example `app/delivery/service.py`) with a dispatcher:
  - `deliver(intent, topic, metadata, user_id, message_text) -> str`
- Integrate into `DevAssistantBrain.handle_message` after intent parsing/logging.
- Keep existing fallback behavior for invalid JSON and internal errors.

### 2) Add persistence for actionable intents
- Extend Python DB schema (`app/database.py`) with new tables:
  - `scheduled_tasks`: reminder/update requests with status and schedule fields.
  - `expense_logs`: amount/currency/category/note/source message.
  - `code_runs` (optional but recommended): language, snippet hash, exit status, runtime, truncated output.
- Add indexes for user + timestamp for efficient retrieval.
- Add DB methods to insert/read each new entity.

### 3) Implement intent handlers (MVP)
- `schedule_task`
  - Parse minimal scheduling metadata from intent payload.
  - Persist task and return confirmation text.
  - If schedule details are incomplete, return a clarification-style response message (still delivery output, not classifier fallback).
- `execute_code`
  - Execute code in a controlled subprocess sandbox (timeout + output limits + allowlist languages).
  - Capture stdout/stderr/exit code and return summarized execution result.
  - Persist run metadata for auditability.
- `debug_code`
  - Use a dedicated debugging prompt to produce diagnosis + likely fix + next step.
  - Prefer deterministic response shape and truncate very long traces before model call.
- `summarize_link`
  - If URL is present: fetch content then summarize.
  - If long text is provided: summarize directly.
  - Return concise bullet-style output.
- `log_expense`
  - Extract/normalize amount + currency + label, persist entry, and confirm logged value.
- `general_chat`
  - Generate normal conversational/dev-help response via a separate chat prompt (not the classifier prompt).

### 4) Split classifier and responder prompts
- Keep current prompt only for classification.
- Add separate prompts/utilities for:
  - conversational response generation,
  - debug analysis,
  - summarization.
- This prevents the current hard JSON prompt from forcing non-user-facing outputs.

### 5) Wire bridge + output formatting
- Ensure `/message` endpoint keeps returning plain text suitable for WhatsApp transport.
- Keep JSON internals in logs/DB only.
- Ensure error responses remain human-readable and consistent.

### 6) Legacy Node mode parity decision
- Preferred implementation order: ship full delivery in Python bridge path first (`start:bridge`).
- Then mirror delivery support into Node legacy mode (`src/index.js`) or clearly mark legacy mode as intent-classification-only.

### 7) Testing plan
- Unit tests (Python):
  - intent validation and dispatch mapping,
  - each handler success + failure path,
  - sandbox timeout/output truncation behavior,
  - DB inserts for tasks/expenses/runs.
- Integration tests:
  - `/message` returns user-facing text for each intent.
  - invalid classifier output falls back safely.
- Regression checks:
  - existing database tests still pass.

### 8) Operational safeguards
- Add execution safety limits:
  - max runtime, memory/output cap, blocked shell operations.
- Add URL fetch limits:
  - timeout, content length cap, unsupported mime handling.
- Add structured logging around delivery decisions and failures.

## Proposed Delivery Order
1. Python delivery dispatcher + response contract.
2. DB schema/methods for tasks and expenses.
3. Implement `schedule_task` and `log_expense` handlers first (lowest risk).
4. Implement `summarize_link`.
5. Implement `debug_code`.
6. Implement `execute_code` as Python-only with strict sandbox constraints.
7. Add tests and finalize bridge behavior.
8. Decide/implement Node legacy parity.

## Confirmed Decisions
- Scope: implement delivery in Python bridge mode first (`start:bridge`), defer Node legacy parity.
- Execute-code MVP: Python-only execution with strict timeout/output/resource limits.

## Acceptance Criteria
- Sending any of the six intents produces a useful WhatsApp reply, not raw JSON.
- Intended side effects are persisted for schedule/expense/code-run flows.
- Failures degrade gracefully with actionable messages.
- Bridge mode works end-to-end with current startup scripts.
- Test suite passes for new and existing tests.

## Risk Notes
- `execute_code` is the highest-risk feature (security and abuse). Implement with strict boundaries first, then expand language support.
- Ambiguous natural-language scheduling may need iterative improvements; MVP should prioritize deterministic confirmations.
